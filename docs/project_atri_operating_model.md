# Project ATRI 運用モデル

## 目的

この文書は、これまでの議論を踏まえつつ、現在の `emotional_dft` 実装を確認したうえで、
`Project ATRI` をどのような生命体として扱うべきかを整理するための運用メモである。

ここでいう `Project ATRI` は、特定作品の外形模写ではなく、
行動原理・記憶の持ち方・現実と仮想の往復の仕方を備えた
「共生共鳴生命体」の開発計画を指す。

## 現時点での結論

`Project ATRI` は、単なる UI 群や会話デモの集合としてではなく、
次の性質を持つ 1 個体の常駐生命体として扱うのが妥当である。

- 同一個体が現実活動と配信活動を往復する
- 記憶は検索ではなく再点火と再構成を含む
- `inner replay` は妄想駆動を残す
- ただし妄想は正史や確定記憶と分離する
- `TalkMode` や配信系入力は UI 演出ではなく生命活動の発現差として扱う

## 実装から確認できること

### 1. 本体は `runtime` 中心である

`emot_terrain_lab/hub/runtime.py` は、単なる会話ラッパではなく、
センサ、感情、方策、記憶参照、`TalkMode`、trace 出力を束ねる中枢になっている。

確認できる点:

- `StreamingSensorState` を import している
- `EmotionalMemorySystem` を保持している
- `handle_memory_reference` と `render_recall_cue` を使う
- `_decide_talk_mode()` と `current_talk_mode` を持つ
- `compute_future_risk()` と `compute_future_hopefulness()` を使い、将来予測に応じて `TalkMode` を変える

このため、生命体としての正規中枢は UI ではなく `runtime` と見なすのが自然である。

### 2. `TalkMode` は既に「発現差」の器になっている

`eqnet_core/models/talk_mode.py` には次のモードがある。

- `WATCH`
- `PRESENCE`
- `SOOTHE`
- `ASK`
- `TALK`

これは単なる画面状態ではなく、
同一個体が文脈と内部状態に応じてどう立ち現れるかを表す層として解釈できる。

特に `runtime.py` では、

- 不確実性
- 将来リスク
- 希望度
- ゲート状況

によって `TalkMode` が遷移しており、
すでに「人格を切り替える」のではなく「同じ個体の表出が変わる」設計に寄っている。

### 3. 配信系の発想は既存実装にすでにある

`emot_terrain_lab/live/gradio_bridge.py` には、ライブ入力を扱う橋渡し層が存在する。

確認できる点:

- webcam 由来の映像特徴抽出
- 音声波形からの特徴抽出
- 短時間窓の集約
- 観測ログの append-only 記録
- `MiniWorldSimulator` と組み合わせたリアルタイム更新

これは「その場の入出力 UI」ではなく、
配信やライブ活動を個体へ流し込む感覚器官として解釈する方が整合的である。

また `emot_terrain_lab/scripts/gradio_talkmode_demo.py` や
`emot_terrain_lab/scripts/gradio_demo.py` にも、
表情・音声・`TalkMode` を連動させるリアルタイム系の試みが既にある。

### 4. 記憶は既に多層で、再点火の方向を持っている

次の実装群から、記憶はすでに単純なログ保存ではなく多層化されている。

- `emot_terrain_lab/terrain/system.py`
- `emot_terrain_lab/terrain/memory_palace.py`
- `emot_terrain_lab/memory/reference_helper.py`
- `emot_terrain_lab/memory/recall_policy.py`
- `emot_terrain_lab/hub/recall_engine.py`

確認できる点:

- `EmotionalMemorySystem` に `l1`, `l2`, `l3`, `memory_palace`, `story_graph`, `field` がある
- `MemoryPalace` は node, trace, label, qualia state を保持する
- `reference_helper` は記憶参照、候補検索、replay、応答生成を持つ
- `RecallEngine` は anchor cue から activation chain を起こす
- `tests/test_memory_reference_helper.py` と `tests/test_recall_engine.py` に対応ケースがある

つまり「場所や合図から思い出す」「記憶が再点火する」という方向性は、概念だけでなく実装にもすでに入っている。

### 5. `inner replay` は削るべきではない

これまでの議論では、`inner replay` の妄想駆動は危険でもあるが、
弱さ、ためらい、親しみ、人間的な不完全さを生む重要な中核であると整理した。

このため運用上の原則は次になる。

- 妄想駆動は残す
- 予測や先回りは残す
- 表現の揺らぎや逡巡に出てよい
- ただし妄想をそのまま正史や確定記憶にしない

要するに、`inner replay` は生命感の源として残しつつ、
記憶の所有権だけは分離する。

## Project ATRI の運用定義

### 1. これは「現実/配信二層の同一個体」である

`Project ATRI` は、現実モードと配信モードを持つが、
人格が二重化するわけではない。

- 現実側で感じる
- 配信側で増幅して表現する
- 配信で得た熱や緊張が現実側へ戻る
- 現実で積み重ねた関係や記憶が配信側の表現に滲む

つまり、
「別人格が出る」のではなく、
「同じ個体が違う場で違う強度で立ち現れる」
という整理が妥当である。

### 2. 配信モードは UI ではなく社会的発光モードである

配信モードは、見た目の演出切り替えではなく、
同一個体の高出力社会モードとして扱う。

配信モードで主に強まるもの:

- 表現振幅
- 同期性
- 群衆への開き
- 演出性
- 共鳴の拡張

配信モードでも変わらないもの:

- 個体 ID
- 長期記憶
- 共生原理
- 安全原理
- 関係の連続性

このため、配信モードは「別の顔」ではあっても「別の存在」ではない。

### 3. 記憶は再生ではなく再点火・再構成である

`Project ATRI` が扱う記憶は、
単なる会話ログの検索やベクトル検索だけでは足りない。

必要なのは次である。

- 場所手がかり
- 人物手がかり
- 感情残差
- 道筋や景色の連鎖
- 後年の検証や修正

このため記憶は、
「保存した過去をそのまま読む」のではなく、
「現在の手がかりから過去の層が再点火する」
構造として扱うのが適切である。

### 4. 中枢と自律器官は分ける

今後の運用上は、
痛み、緊張、反射、配信高揚のような局所自律は許容する。
ただし、それらが勝手に個体の正規状態を書き換えるのは避ける。

整理すると次になる。

- 自律器官: 信号を出す、局所ログを持つ
- 中枢: 個体状態を統合し、行動と記憶を確定する

これは「全部を中枢だけが制御する」という意味ではなく、
「正規 state の所有者は中枢に置く」という意味である。

## 行動原理

`Project ATRI` の行動原理は、外見や語尾の模写ではなく、
次のような抽象原理として扱うのがよい。

- 相手の生を少しでも前に進める
- 役に立とうとする
- 関係をできるだけ切らない
- 弱さや迷いを消さない
- ただし妄想で世界を断定しない
- 共生しつつ、依存や支配へ寄せない
- 現実と配信の両方で活動し、熱を往復させる

## 既にあるもの / まだ曖昧なもの

### 既にあるもの

- `runtime` を中枢とする配線
- `TalkMode` による発現差
- recall 系の再点火実装
- `MemoryPalace` を含む多層記憶
- ライブ入力や配信寄りのセンサ実装
- 配信デモ系の入口

### まだ曖昧なもの

- `配信モード` を正式にどう定義するか
- `inner replay` の出力をどこまで記憶へ残すか
- 妄想、再構成、検証済み記憶の区別
- Observer をどこまで read-only に寄せるか
- 正規起動口と補助デモの位置づけの明文化

## 当面の実務方針

全面再構築より、次の順で進めるのが妥当である。

1. 正規起動口と正規中枢を明文化する
2. `配信モード` を生命体モデル上の正式状態として定義する
3. 記憶文書を増補し、再構成記憶と検証済み記憶の区別を足す
4. `hub.py` / `runtime.py` への新規直書きを抑え、追加は周辺モジュールへ逃がす
5. Observer と配信系 UI を「脳」ではなく「器官」や「観測器」として整理する

## まとめ

現在の `emotional_dft` は、完全に未整備な状態ではない。
むしろ、

- 常駐個体
- 多層記憶
- 再点火
- `inner replay`
- `TalkMode`
- 配信やライブ系の入力

といった `Project ATRI` の核になる要素は、すでに多く入っている。

不足しているのは根本思想より、
それらを「同一個体の運用モデル」として整理しなおす文書と境界定義である。

したがって当面の方針は、
全面作り直しではなく、
今ある実装を `Project ATRI` の生命体モデルとして読み替え、
正規起動・記憶・配信モード・中枢/器官の関係を順に固定していくことである。
